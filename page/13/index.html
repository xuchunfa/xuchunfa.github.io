  • 阻塞队列

    阻塞队列会对当前线程产生阻塞，比如一个线程从一个空的阻塞队列中取元素，此时线程会被阻塞直到阻塞队列中有了元素。
当队列中有元素后，被阻塞的线程会自动被唤醒（不需要我们编写代码去唤醒）。这样提供了极大的方便性。

  • wait()、notify()和notifyAll()

    1.如果调用某个对象的 wait()/notify() 方法，当前线程必须拥有这个对象的monitor（即锁），因此调用 wait()/notify()
方法必须在同步块或者同步方法中进行（synchronized块或者synchronized方法)。

2.调用某个对象的wait()方法，相当于让当前线程交出此对象的monitor，然后进入等待状态，等待后续再次获得此对象的锁。（而sleep不会释放该对象的锁）

3.notifyAll() 唤醒所有正在等待该对象的monitor的线程。

4.一个线程被唤醒不代表立即获取了对象的monitor，只有等调用完notify()或者notifyAll()并退出synchronized块，释放对象锁后，其余线程才可获得锁执行。

  • Condition

它用来替代传统的Object的wait()、notify()实现线程间的协作,更加安全和高效。

1.Condition依赖于Lock接口，生成一个Condition的基本代码是lock.newCondition()。

2.调用Condition的await()和signal()方法，都必须在lock保护之内，就是说必须在lock.lock()和lock.unlock之间才可以使用。

3.
    Conditon中的await()对应Object的wait()；
    Condition中的signal()对应Object的notify()；
    Condition中的signalAll()对应Object的notifyAll()。

  • CyclicBarrier

CyclicBarrier强调的是n个线程，大家相互等待，只要有一个没完成，所有人都得等着。

重要字段：

private final int parties;//需要相互等待的线程数

private final Runnable barrierCommand;

private Generation generation = new Generation();

private int count;//需要等待的线程的数量

进入：

private int dowait(boolean timed, long nanos)
    throws InterruptedException, BrokenBarrierException,
           TimeoutException {
    final ReentrantLock lock = this.lock;
    lock.lock();
    try {
        final Generation g = generation;

        if (g.broken)
            throw new BrokenBarrierException();

        if (Thread.interrupted()) {
            breakBarrier();
            throw new InterruptedException();
        }

        int index = --count;
        if (index == 0) {  // tripped
            boolean ranAction = false;
            try {
                final Runnable command = barrierCommand;
                if (command != null)
                    command.run();//调用构造函数中的Runnable实现
                ranAction = true;
                nextGeneration();//唤醒所有阻塞的线程
                return 0;
            } finally {
                if (!ranAction)
                    breakBarrier();
            }
        }


        for (;;) {
            try {
                if (!timed)
                    trip.await();//线程还没到齐就一直阻塞
                else if (nanos > 0L)
                    nanos = trip.awaitNanos(nanos);
            } catch (InterruptedException ie) {
                if (g == generation && ! g.broken) {
                    breakBarrier();
                    throw ie;
                } else {
                    // We're about to finish waiting even if we had not
                    // been interrupted, so this interrupt is deemed to
                    // "belong" to subsequent execution.
                    Thread.currentThread().interrupt();
                }
            }

            if (g.broken)
                throw new BrokenBarrierException();

            if (g != generation)
                return index;

            if (timed && nanos <= 0L) {
                breakBarrier();
                throw new TimeoutException();
            }
        }
    } finally {
        lock.unlock();
    }
}

看一眼唤醒的方法：

private void nextGeneration() {
    // signal completion of last generation
    trip.signalAll();
    // set up next generation
    count = parties;
    generation = new Generation();
}

单例模式的不同写法

  • 饿汉模式（在调用对象之前就已经初始化了）

    1.基于类加载的机制（是线程安全的！！！）

    public class Singleton {

    private static final Singleton INSTANCE = new Singleton();

    private Singleton(){}

    public static Singleton getInstance(){

    return INSTANCE;

    }
    }

这里加final的意思这么理解：

final关键字可以禁止指令重排，防止在 INSTANCE 还没完全赋值的情况下被使用.

为什么线程安全参考静态内部单例模式。

  • 懒汉模式（线程不安全）

       public class Singleton {  
  private static Singleton instance;  
  private Singleton (){
  }   
  public static Singleton getInstance() {  
  if (instance == null) {  
          instance = new Singleton();  
  }  
  return instance;  
  }  
}

这里为什么不能加final:

1.因为final static 域必须要（直接赋值）或者（通过静态代码块）来赋值，这样就直接初始化了，没有做到延迟初始化。

2.如果你是final非static成员，必须在构造器、代码块、或者直接定义赋值；

  • 懒汉模式（线程安全）

    1.同步开销大,效率不高。

  • 双重检查模式（DCL）

    public class Singleton { 

    private static volatile Singleton instance;

    private Singleton(){}

    public static Singleton getInstance(){
        if(instance == null){
            synchronied(Singleton.class){
                if(instance == null){
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }

  • 静态内部单例模式

    public class Singleton { 
private Singleton(){
    }
  public static Singleton getInstance(){  
    return SingletonHolder.sInstance;  
}  
private static class SingletonHolder {  
    private static final Singleton sInstance = new Singleton();  
}  
}

1.第一次加载Singleton类不会实例化对象,只有在调用getInstance()方法时才会引起SingletonHolder类加载的理解？？？

1.1 外部类 Singleton 加载的时候静态内部类 SingletonHolder 不会加载，内部类的加载不需要依附外部类，在使用时才会加载。所以当调用内部类 SingletonHolder 的静态域 sInstance 才会引起内部类的加载以及初始化。

1.2 不过当只调用外部类的静态变量，静态方法时，是不会让静态内部类的被加载。

2.为什么是线程安全的？？？

2.1 首先解释一下clinit()方法中包含类中的所有类（static）变量的赋值动作和静态语句块。

2.2 虚拟机会保证一个类的类构造器clinit()在多线程环境中被正确的加锁、同步，如果多个线程同时去初始化一个类，那么只会有一个线程去执行这个类的类构造器clinit()，其他线程都需要阻塞等待，直到活动线程执行clinit()方法完毕。

特别需要注意的是，在这种情形下，其他线程虽然会被阻塞，但如果执行clinit()方法的那条线程退出后，其他线程在唤醒之后不会再次进入执行clinit()方法，因为 在同一个类加载器下，一个类型只会被初始化一次。如果在一个类的clinit()方法中有耗时很长的操作，就可能造成多个线程阻塞，在实际应用中这种阻塞往往是隐藏的.

一句话概括：java虚拟机保证在多线程的环境下类能够被正确加载。
  • 枚举单例模式

枚举类型的好处

单例模式是如何保证单例的？

即怎么保证内存中只有一个对象：

1.外界不能随意创建对象，所有构造函数必须私有化；

2.类本身要有一个对象。即类里面要封装好一个对象，下次通过公共方法调用时都是返回的这个对象；

3.通过公共的方法提供给别人。因为外界不能创建对象，所以通过给方法加static关键字，通过类名访问类方法来获取对象；

  • instance为什么要加static关键字？

    因为单例模式中构造器被私有化，产生实例的唯一途径就是通过 static 方法，
在类加载的时候创建对象，这样就保证对象的唯一性。。。
而static方法中只能访问静态变量,所以必须声明成static.

  • 单例模式中构造函数为什么私有化？

    1.构造函数私有化后，那么只能通过这个类的内部函数才能构造这个类的对象。

2.将生成类对象的方法用static修饰的原因是：因为静态方法不需要依赖实例化的对象。

  • 单例模式中volatile变量的作用？

    保证原子性和有序性；

    首先分析 instance = new Singleton() 的本质：

    1.为对象分配内存空间

    2.对象初始化

    3.将对象的引用指向对应的内存地址。

但是在单线程内2和3可能会发生重排序，于是发生了如下情况：

此时线程B访问的是没有经过初始化的对象，会报错，所以使用volatile的根源在于：禁止A2和A3的重排序。

  • 单例模式中为什么要使用双重检查索(DCL)？

    因为在多线程的环境下，所有线程在调用getSingleton()方法时都必须在synchronized这里排队，
而加锁是一个非常耗时的操作，这导致单例模式的效率非常低。
我们只需要在实例还没有创建之前添加加锁操作。

  • 静态内部类方法创建单例模式的好处？

    按需创建实例（饿汉式模式：在类加载时就完成了初始化），提高了内存的利用效率。

    这种方式的特点是：Singleton类被装载了，instance不一定被初始化，只有显示通过调用getInstance方法时，才会显示装载SingletonHolder类，从而实例化instance。`

  • 栈内存和堆内存的比较？

**堆内存存的内容：**数组元素、类对象、类中的非静态成员变量（实例变量）、包装类数据(Integer等)、

**栈内存内容：**

1.方法中声明的变量(因为方法执行时，被分配的内存就在栈中,即方法栈)：分基本数据类型变量和引用变量，前者的变量名和值都存在方法栈中，后者声明的变量存放在栈中，所指的对象放在堆中。

2.对象的引用

**方法区：**线程共享的内存区域。类里面的静态成员和静态方法、

**常量池：**方法区的一块内存。

**静态域：**方法区的一块内存。存放类中的静态成员变量

  • 局部变量保存在栈内存中，随方法的消失而消失;成员变量存储在堆中的对象里，由垃圾回收器负责回收。

  • 包装类数据Integer、String等等存在于堆中。

  • 字符串不可变的原因？

需要考虑的临界条件

  • 怎么设置limit?

    错误：

    正号时：limit = Integer.MAX_VALUE；

    负号时：limit = -Integer.MIN_VALUE；

    条件：result*10 + digit > limit

    计算：result = result*10 + digit

    limit的类型为int,越界了！！！

正确：

正号时：limit = -Integer.MAX_VALUE；

负号时：limit = Integer.MIN_VALUE；

条件：result*10 - digit < limit 为true时 越界

计算：result = result*10 - digit

    例如：str = “21474836471”

    方法：

    int flag = limit/radix;

    if(result < flag){

public class StringToInteger {

public static int vary(String s) {

    int i = 0;
    int len = s.length();
    int result = 0;
    int radix = 10;
    boolean negative = false;//默认正数
    int limit = -Integer.MAX_VALUE;//-2147483647

   /* if(s == null){
        System.out.println("null");
        return -1;
    }*/

    if (s == null || s.length() == 0) {
        System.out.println("字符串不能为空!");
        return -1;
    }

    if (len > 0) {
        char firstChar = s.charAt(i);

        //判断第一个字符是不是正负号

            if (firstChar < '0') {
            if (firstChar == '-') {
                limit = Integer.MIN_VALUE;//-2147483648
                negative = true;
            } else if (firstChar != '+') {
                System.out.println("非法字符!");
                return -1;
            }

            if (len == 1) {
                System.out.println("非法格式");
                return -1;
            }
            i++;
        }

        //用于判断result < Integer.MIN_VALUE 的情况!!!
        int flag = limit/radix;


        //接着开始循环
        while (i < len) {
            char ch = s.charAt(i);
            if (ch >= '0' && ch <= '9') {
                int digit = ch - '0';

                if(result < flag){
                    System.out.println("越界!");
                    return -1;
                }
                //这里result定义的为int型,有可能越界
                result = result * radix;
                if (result - digit < limit) {
                    System.out.println("越界!");
                    return -1;
                }
                result -= digit;

            } else {
                System.out.println("非法字符!");
                return -1;
            }
            i++;
        }
    }
    //结束 if(len > 0)
    return negative ? result : -result;
}

public static void main(String[] args) {
    String s = "";
    //String s = "";

    //String s = "2147483648";

    //String s = "-1";
    //String s = "-2147483648";
    //String s = "-2147483649";
    //String s = "-2147483641";

}

Github常用命令总结

发表于 2018-07-04 | 分类于 Github | 阅读次数:

  • master: 本地项目
  • origin: Fork的远程项目
  • upstream: 别人的开源项目

git pull = git fetch + git merge

例如：
git pull origin next
等价于

` git fetch origin`
 `git merge origin/next`

常用命令图解：

  • 代码提交

    例如：提交工作区自上次commit之后的变化，直接到仓库区

    git commit -a

    例如：提交暂存区到仓库区

    git commit -m [message]

  • 切换分支

    例如：切换到指定分支，并更新工作区

    git checkout [branch-name]

  • 合并分支

    例如：合并指定分支到当前分支

    git merge [branch]

  • 查看远程仓库

    `git remote -v `

  • 添加远程仓库

    `git remote add pb https://github.com/paulboone/ticgit`

  • 从远程仓库中抓取与拉取

    git fetch [remote-name]

  • 推送到远程仓库

    git push [remote-name] [branch-name]

  • 在origin/master基础上创建一个新分支

    git checkout -b newBranch origin/master

  • 在本地分支上合并远程分支

    git merge origin/master

    或者

    git rebase origin/master

  • 取回远程主机某个分支的更新，再与本地的指定分支合并

    git pull <远程主机名> <远程分支名>:<本地分支名>

    例如：

    `git pull origin next:master`

    远程分支与当前分支合并：

    `git pull origin next`

  • 将本地分支更新，推送到远程主机

    git push <远程主机名> <本地分支名>:<远程分支名>

    不要弄混淆了：

    pull 拉取远程分支名

push 推送本地分支名

    例如：将本地的branch1分支推送到origin主机的master分支

    `git push origin branch1`
