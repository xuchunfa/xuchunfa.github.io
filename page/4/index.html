  1. 进程和线程的区别？

    1.进程是资源（CPU、内存、磁盘等等）分配的基本单位，线程是调度（程序执行的）的基本单位。进程是是实实在在看得见的，Ctrl+Shift+Esc显示进程。

2.同一进程的各个线程共享该进程的内存资源。但是当一个线程使用某些共享内存时，其他线程必须等待。

3.进程有自己的独立地址空间，同一进程里的线程享有相同的地址空间；进程切换代价昂贵，线程切换代价小。

4.多进程程序更强壮，多线程程序只要有一个线程挂了，进程就死掉了，但不会影响其他进程。

  1. 进程的通信方式？

    1.管道：半双工、数据单向流动、（无名管道）具有亲缘关系（父子进程）的进程间。其中命名管道允许不具有亲缘关系的进程通信。

    1.1  fork()创建子进程，若成功返回子进程的pid,失败返回-1；

    1.2 子进程会复制父进程的几乎所有信息；

2.信号量：控制多个进程访问共享资源。

    2.1 Linux内核中提供了三种信号量：二元信号量（互斥信号量）、计数信号量和读者写者信号量；

    2.2 读者写者信号量：允许多个并发的读者（使用计数信号量），但仅允许一个写者（使用互斥信号量）。

3.消息队列：克服了信号承载信号量小，管道只能承载无格式字节流以及缓冲区大小受限的缺点。
4.信号：通知接受事件发生。
5.共享内存：最快IPC通信，因为数据不需要在不同的进程间复制，通常信号量被用来对共享数据的存储同步；
6.套接字（socket）：可用于不同机器间的进程通信。

5.关于concurrentHashMap的理解

concurrentHashMap JDK1.7和JDK1.8的理解

注意：

1.JDK1.8 和JDK1.7 concurrentHashMap不允许key或value为null
  当你通过get(k)获取对应的value时，如果获取到的是null时，你无法判断，它是put（k,v）的时候value为null，还是这个key从来没有做过映射;

2.JDK1.7 concurrentHashMap的每个Segment中含有一个
  transient volatile HashEntry<K,V>[]table，保证了entryAt/setEntryAt的可见性；

总结：

1.JDK1.8中的concurrentHashMap大量使用CAS算法，即无锁化的修改值的操作。算法思想：比较当前内存中的变量值与指定的expect变量的值是否相等，相等则修改expect变量的值为new value.

2.JDK1.8 concurrentHashMap的put操作：

    2.1 通过casTabAt(tab, i, null,new Node<K,V>(hash, key, value, null)方法中的

        U.compareAndSwapObject(tab, ((long)i << ASHIFT) + ABASE, c, v)CAS方法

        在表头Node<K,V>[] table中插入节点；

    2.2 采用synchronized + CAS的思想。synchronized锁住的是hash值相等的链表tab[i]的头节点；

    2.3 采用尾插法插入新节点；

3.JDK1.8 concurrentHashMap的get操作：

    3.1 读操作没有加锁。key的hash值小于0就去红黑树中查找；hash值大于0就去链表中查找。

    3.2 利用volatile线程可见性 (Node<K,V>)U.getObjectVolatile(tab,i);


4.JDK1.7 concurrentHashMap的put操作：

    4.1 获取segment的位置后尝试获取锁，获取到锁后node=null;没获取到锁的线程则通过scanAndLockForPut尝试获取锁；

    4.2 scanAndLockForPut的思想是：通过自旋不断获取segment的锁，并在获取的过程中遍历链表，

        没有找到和key equals的HashEntry则创建一个HashEntry并在获取到锁后返回它，当尝试次数超过

        最大尝试次数MAX_SCAN_RETRIES则加锁等待。

    4.3 因为插入节点采用的头插法，所以在scanAndLockForPut会判断first有无发生变化，发生变化（有线

        程加入节点）则重新遍历链表；


5.JDK1.7 concurrentHashMap的get操作：

    5.1 读操作没有加锁。通过volatile读语义保证可见性。只有当put和remove调用
    setEntryAt（table,i）方法后（已经释放锁），读操作才能读到变化的table;

    5.2 通过UNSAFE.getObjectVolatile(table,i)获得Segment<K,V>或HashEntry<K,V>[]的表头节点

    保证了线程之间的可见性；

  3. 死锁的产生的原因以及怎么预防？

    原因：

 1.系统资源不足；

 2.进程运行的推进顺序不合理；

 3.资源分配不当；

    四个必要条件：

    1.互斥条件：一个资源每次只能被一个进程使用；

2.请求和保持条件：一个进程因请求资源而阻塞时，对已经获得的资源保持不放；

3.不剥夺条件：进程已经获得的资源，在未使用之前不能强行剥夺；

4.循环等待条件：若干进程之间形成了一种头尾相接循环等待资源的关系；

    预防死锁：

    1.破环请求和保持条件：一次性申请所需要的全部资源；

2.破坏不剥夺条件：直接抢占其他进程的资源，适用于状态易于保存和恢复的资源；

3.破环循环等待条件：资源排序，得到低优先级的资源后才能申请高优先级的资源，高优先级的资源不允许访问低优先级的资源；

    解决死锁的策略：

    1.死锁预防：破坏四个必要条件；

2.死锁避免：

    2.1 如果一个进程的请求会导致死锁，则不启动此进程；

    2.2 指进程每次申请资源时判断安不安全，例如银行家算法。

3.死锁检测：事后措施；

3.根据锁的状态分为：

    3.1 对象头的介绍：

        1.Mark Word:存储对象的hashcode、分代年龄、和锁标志位信息；

        2.klass point:对象指向它的类元数据的指针；

        目的：出现不同锁状态的原因，都是为了减少获得锁和释放锁带来的性能消耗，

    3.2 无锁（01）：CAS应用；

    3.3 偏向锁（01）：指一段同步代码一直被一个线程访问，那么该线程会自动获取锁，降低获取锁的代价;

        3.3.1 实现方法：即检测 Mark Word里是否存储着指向当前线程的偏向锁；

        3.3.2 好处：为了在无多线程竞争的情况下尽量减少不必要的轻量级锁执行路径，因为轻量级锁的获取及释放依赖多次CAS原子指令，
        而偏向锁只需要在置换ThreadID的时候依赖一次CAS原子指令即可

        3.3.3 偏向锁的释放：偏向锁只有遇到其他线程尝试竞争偏向锁时，持有偏向锁的线程才会释放锁，线程不会主动释放偏向锁。偏向锁的撤销，
        需要等待全局安全点（在这个时间点上没有字节码正在执行），它会首先暂停拥有偏向锁的线程，判断锁对象是否处于被锁定状态。
        撤销偏向锁后恢复到无锁（标志位为“01”）或轻量级锁（标志位为“00”）的状态；

    3.4 轻量级锁（00）：当锁是偏向锁时被其他线程访问，锁升级为轻量级锁，其他线程将会通过自旋尝试获取锁，不阻塞，提高性能；

        3.4.1 具体细节见下：

            在代码进入同步块的时候，如果同步对象锁状态为无锁状态（锁标志位为“01”状态，是否为偏向锁为“0”），虚拟机首先将在当前线程的栈帧中建立一个名为锁记录（Lock Record）的空间，
            用于存储锁对象目前的Mark Word的拷贝，然后拷贝对象头中的Mark Word复制到锁记录中。

            拷贝成功后，虚拟机将使用CAS操作尝试将对象的Mark Word更新为指向Lock Record的指针，并将Lock Record里的owner指针指向对象的Mark Word。

            如果这个更新动作成功了，那么这个线程就拥有了该对象的锁，并且对象Mark Word的锁标志位设置为“00”，表示此对象处于轻量级锁定状态。

            如果轻量级锁的更新操作失败了，虚拟机首先会检查对象的Mark Word是否指向当前线程的栈帧，如果是就说明当前线程已经拥有了这个对象的锁，那就可以直接进入同步块继续执行，否则说明多个线程竞争锁。

            若当前只有一个等待线程，则该线程通过自旋进行等待。但是当自旋超过一定的次数，或者一个线程在持有锁，一个在自旋，又有第三个来访时，轻量级锁升级为重量级锁。


    3.5 重量级锁（10）：等待锁的线程都会进入到阻塞状态；

    偏向锁通过对比Mark Word解决加锁问题，避免执行CAS操作。而轻量级锁是通过用CAS操作和自旋来解决加锁问题，
    避免线程阻塞和唤醒而影响性能。重量级锁是将除了拥有锁的线程以外的线程都阻塞。

class Father implements Cloneable{
    private String fatherName;
    private Child child;

    public Father(String fatherName,Child child) {
        this.fatherName = fatherName;
        this.child = child;
    }

    @Override
    protected Object clone() throws CloneNotSupportedException {
        //重写clone方法
        Father father = (Father) super.clone();
        father.child = (Child) father.child.clone();
        return father;
    }
}

class Child implements Cloneable{
    private String childName;

    public Child(String childName) {
        this.childName = childName;
    }

    @Override
    protected Object clone() throws CloneNotSupportedException {
        return super.clone();
    }
}

@Test
public void test() throws CloneNotSupportedException {
    Child child = new Child("xiaoxu");
    Father father = new Father("daxu",child);
    Father clonedFather = (Father) father.clone();
    System.out.println(father.child == clonedFather.child);//false
}

    如果一个对象内部只有基本数据类型，那用 clone() 方法获取到的就是这个对象的深拷贝，

代理模式

发表于 2018-11-08 | 分类于 Java设计模式 | 阅读次数:

总结：

1.代理模式优点：

1.良好的拓展性。修改被代理的角色不影响调用者使用代理；

2.隔离，降低了耦合度。被代理的角色只需关注业务逻辑，通过代理角色可以处理和隔离非业务操作；

2.代理模式与装饰者模式区别：

装饰者模式是"新增行为"，代理模式是"控制访问"；

3.Spring AOP

如果加入容器的目标对象有接口实现，使用JDK动态代理；

如果加入容器的目标对象没有实现接口，使用Cglib代理；

静态代理

指在程序运行前就已经存在代理类的字节码文件了，即代理类和委托类关系在运行前就已经确定了；

经纪人和演员的例子：（静态代理）

//Subject角色
public interface Subject {
    void perform();
}

//RealSubject角色
public class Actor implements Subject {
    @Override
    public void perform() {
        System.out.println(getClass().getSimpleName() + ": 负责演戏");
    }
}

//Proxy角色
public class Agent implements Subject {

    private Subject actor;

    public Agent(Subject actor) {
        this.actor = actor;
    }

    @Override
    public void perform() {
        System.out.println(getClass().getSimpleName() + ": 负责接戏");
        actor.perform();
    }
}

Client:

Subject actor = new Actor();
Agent proxy = new Agent(actor);
proxy.perform();

Java动态代理

1.代理类是动态生成的；

//java.lang.reflect.Proxy 返回值为代理类的实例
public static Object newProxyInstance(ClassLoader loader,
                                  Class<?>[] interfaces,
                                  InvocationHandler h)

Proxy.newProxyInstance()内部实现原理过程：

//内部包含委托类的引用target，实现代理类到委托类的分派转发
1.InvocationHandler handler = new InvocationHandlerImpl(..);

//动态创建代理类的类对象
2.Class clazz = Proxy.getProxyClass(classLoader, new Class[] { Interface.class, ... });

//反射获取代理类类对象的构造函数对象
3.Constructor constructor = clazz.getConstructor(new Class[] { InvocationHandler.class }); 

//动态创建代理类的实例
4.Interface Proxy = (Interface)constructor.newInstance(new Object[] {handler});

2.第三个参数由用户实现InvocationHandler接口，重写invoke()方法，实现对被代理对象的方法访问；

public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
    private Animal animal;
    System.out.println("--------start------------");
    Object proxyObject = method.invoke(animal,args);
    System.out.println("---------end----------");
    System.out.println();

    //返回代理对象
    return proxyObject;
}

动物实例：

public interface Animal {

    public void eat(String food);
    public void sleep(String address);

}

//被代理类
public class PriorHuman implements Animal {

    public void eat(String food) {
        System.out.println("PriorHuman eat " + food);
    }

    public void sleep(String address) {
        System.out.println("PriorHuman sleep at " + address);
    }
}

//用户自己的调用处理器
public class InvocationHandlerImpl implements InvocationHandler {

    //更具有通用性
    private Object target;

    public InvocationHandlerImpl(Animal animal) {
        this.target = animal;
    }

    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {

        System.out.println("--------start------------");
        Object proxyObject = method.invoke(target,args);
        System.out.println("---------end----------");
        System.out.println();

        //返回代理对象
        return proxyObject;
    }
}

Client:

public static void main(String[] args) throws Throwable {

    Animal human = new PriorHuman();

    //三个参数
    ClassLoader classLoader = human.getClass().getClassLoader();
    Class[] interfaces = human.getClass().getInterfaces();
    InvocationHandler handler = new InvocationHandlerImpl(human);

    //代理类实例
    Animal proxy = (Animal) Proxy.newProxyInstance(classLoader,interfaces, handler);

    //实际执行过程是通过反射调用委托类的方法
    proxy.eat("hotdog");
    proxy.sleep("Big house");
}

Cglib代理（子类代理）

被代理的对象只是一个单独的对象，没有实现任何接口，这时在内存中构建一个子类对象，实现对

目标对象功能的拓展；

注意：

final类不会被代理，final或static方法不会被拦截，即不会执行目标对象的额外业务方法；

实例：

//被代理的对象
public class UserDao {
    public void save(){
        System.out.println("保存数据.....");
    }
}

//CGLIB代理工厂
public class ProxyFactory implements MethodInterceptor {

    private Object target;
    public ProxyFactory(Object target) {
        this.target = target;
    }

    //给目标对象创建一个代理对象
    public Object getProxyInstance(){
        Enhancer enhancer = new Enhancer();
        //设置父类
        enhancer.setSuperclass(target.getClass());
        //设置回调函数
        enhancer.setCallback(this);

        //创建子类对象(代理对象)
        return enhancer.create();
    }

    //类似于InvocationHandler的作用
    @Override
    public Object intercept(Object o, Method method, Object[] args, MethodProxy methodProxy) throws Throwable {
        System.out.println("开启事务");
        Object result = method.invoke(target,args);
        System.out.println("提交事务");
        return result;
    }
}

Client:

public static void main(String[] args){
    UserDao user = new UserDao();
    UserDao proxy = (UserDao) new ProxyFactory(user).getProxyInstance();
    proxy.save();
}
