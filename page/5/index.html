1.经常增删改的列不需要建立索引；

3.表的记录数不够多；

索引失效的情况：

1.“%aaa%” 不会使用索引,而like “aaa%”可以使用索引；
   使用正则表达式，只有在第一个字符不是通配符的情况下索引才生效；

2.在列上进行运算将会使索引失效，包括计算表达式、数据库函数；

3.MySQL查询只使用一个索引，因此如果where子句中已经使用了索引的话，那么order by中的列是不会使用索引的。

    使用联合索引自动排序的好处：(a,b)

    eg：select ... from table where a=XXX order by b；

4.联合索引未使用最左匹配原则，例如组合索引（A，B），where B=b不会使用索引；

5.只要列中包含有NULL值都将不会包含在索引中，组合索引只要有一列含有NULL
  这一列对于组合索引就是无效的；

6.索引字段使用 !=或<>、is null或is not null、or、not in等将会使索引失效导致全表扫描；

7.索引字段是字符串，但查询的时候不加单引号，索引失效；

static class PrintOddAndEven implements Runnable{
    private static final int NUMBERS = 20;
    private static volatile int index = 1;
    private int flag;
    static Object lock = new Object();//static

    public PrintOddAndEven(int flag) {
        this.flag = flag;
    }

    @Override
    public void run(){
        while(index < NUMBERS){
            synchronized (lock){
                while ((index & 1) != flag){
                    try {
                        lock.wait();
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }
                }
                System.out.println(Thread.currentThread().getName()+": "+index);
                index++;
                lock.notifyAll();//将等待队列中的所有线程移到同步队列中去
            }
        }
    }
}

public static void main(String[] args){
    Thread oddThread = new Thread(new PrintOddAndEven(1));
    Thread evenThread = new Thread(new PrintOddAndEven(0));
    oddThread.start();
    evenThread.start();
}

  3. 死锁的产生的原因以及怎么预防？

    原因：

 1.系统资源不足；

 2.进程运行的推进顺序不合理；

 3.资源分配不当；

    四个必要条件：

    1.互斥条件：一个资源每次只能被一个进程使用；

2.请求和保持条件：一个进程因请求资源而阻塞时，对已经获得的资源保持不放；

3.不剥夺条件：进程已经获得的资源，在未使用之前不能强行剥夺；

4.循环等待条件：若干进程之间形成了一种头尾相接循环等待资源的关系；

    预防死锁：

    1.破环请求和保持条件：一次性申请所需要的全部资源；

2.破坏不剥夺条件：直接抢占其他进程的资源，适用于状态易于保存和恢复的资源；

3.破环循环等待条件：资源排序，得到低优先级的资源后才能申请高优先级的资源，高优先级的资源不允许访问低优先级的资源；

    解决死锁的策略：

    1.死锁预防：破坏四个必要条件；

2.死锁避免：

    2.1 如果一个进程的请求会导致死锁，则不启动此进程；

    2.2 指进程每次申请资源时判断安不安全，例如银行家算法。

3.死锁检测：事后措施；

4.死锁解除：强行剥夺进程的资源；

9.CAS算法出现的问题？

1.CAS机制

    compareAndSwapInt(内存值、预期值、新值)，当更新时比较内存值和预期值是否相等
    相等的话则更新，否则一直自旋重试；

2.问题

    1.ABA问题。

     在变量前面加上版本号，每次变量更新的时候版本号加1，变化过程从A-B-A变成了1A-2B-3A了。

    2.循环时间开销大。CAS操作长时间不成功的话会一直自旋，CPU开销大。

    3.只能保证一个共享变量的原子操作。

10.什么叫自旋锁？

自旋锁：

    为了降低阻塞和唤醒线程对性能造成的影响，可以让请求锁失败的线程自旋，即不放弃CPU的执行时间，
    当前面的锁释放了同步资源以后，自旋线程就能不必阻塞直接获取同步资源，从而避免了线程切换的开销。

特点：

    自旋时间过长影响整体性能，过短达不到延迟阻塞的目的；

        1.Mark Word:存储对象的hashcode、分代年龄、和锁标志位信息；

        2.class point:对象指向它的类元数据的指针；

        目的：出现不同锁状态的原因，都是为了减少获得锁和释放锁带来的性能消耗，

    3.2 无锁（01）：CAS应用；

    3.3 偏向锁（01）：指一段同步代码一直被一个线程访问，那么该线程会自动获取锁，降低获取锁的代价;

        3.3.1 实现方法：即检测 Mark Word里是否存储着指向当前线程的偏向锁；

        3.3.2 好处：为了在无多线程竞争的情况下尽量减少不必要的轻量级锁执行路径，因为轻量级锁的获取及释放依赖多次CAS原子指令，
        而偏向锁只需要在置换ThreadID的时候依赖一次CAS原子指令即可

        3.3.3 偏向锁的释放：偏向锁只有遇到其他线程尝试竞争偏向锁时，持有偏向锁的线程才会释放锁，线程不会主动释放偏向锁。偏向锁的撤销，
        需要等待全局安全点（在这个时间点上没有字节码正在执行），它会首先暂停拥有偏向锁的线程，判断锁对象是否处于被锁定状态。
        撤销偏向锁后恢复到无锁（标志位为“01”）或轻量级锁（标志位为“00”）的状态；

    3.4 轻量级锁（00）：当锁是偏向锁时被其他线程访问，锁升级为轻量级锁，其他线程将会通过自旋尝试获取锁，不阻塞，提高性能；

        3.4.1 具体细节见下：

            在代码进入同步块的时候，如果同步对象锁状态为无锁状态（锁标志位为“01”状态，是否为偏向锁为“0”），虚拟机首先将在当前线程的栈帧中建立一个名为锁记录（Lock Record）的空间，
            用于存储锁对象目前的Mark Word的拷贝，然后拷贝对象头中的Mark Word复制到锁记录中。

            拷贝成功后，虚拟机将使用CAS操作尝试将对象的Mark Word更新为指向Lock Record的指针，并将Lock Record里的owner指针指向对象的Mark Word。

            如果这个更新动作成功了，那么这个线程就拥有了该对象的锁，并且对象Mark Word的锁标志位设置为“00”，表示此对象处于轻量级锁定状态。

            如果轻量级锁的更新操作失败了，虚拟机首先会检查对象的Mark Word是否指向当前线程的栈帧，如果是就说明当前线程已经拥有了这个对象的锁，那就可以直接进入同步块继续执行，否则说明多个线程竞争锁。

            若当前只有一个等待线程，则该线程通过自旋进行等待。但是当自旋超过一定的次数，或者一个线程在持有锁，一个在自旋，又有第三个来访时，轻量级锁升级为重量级锁。


    3.5 重量级锁（10）：等待锁的线程都会进入到阻塞状态；

        由对象内部的 Monitor 实现，本质是依赖于底层操作系统的 Mutex Locks实现
        实现线程切换需要实现用户态和内核态的切换，代价高；

总结：

    偏向锁通过对比Mark Word解决加锁问题，避免执行CAS操作。

    而轻量级锁是通过用CAS操作和自旋来解决加锁问题，避免线程阻塞和唤醒而影响性能。

    重量级锁是将除了拥有锁的线程以外的线程都阻塞。

class Father implements Cloneable{
    private String fatherName;
    private Child child;

    public Father(String fatherName,Child child) {
        this.fatherName = fatherName;
        this.child = child;
    }

    @Override
    protected Object clone() throws CloneNotSupportedException {
        //重写clone方法
        Father father = (Father) super.clone();
        father.child = (Child) father.child.clone();
        return father;
    }
}

class Child implements Cloneable{
    private String childName;

    public Child(String childName) {
        this.childName = childName;
    }

    @Override
    protected Object clone() throws CloneNotSupportedException {
        return super.clone();
    }
}

@Test
public void test() throws CloneNotSupportedException {
    Child child = new Child("xiaoxu");
    Father father = new Father("daxu",child);
    Father clonedFather = (Father) father.clone();
    System.out.println(father.child == clonedFather.child);//false
}

        1.主要是对Eden区进行GC，会发生Stop The World事件、同时会计算Eden和Survivor的大小

    7.2 全局并发标记(global concurrent marking)

        1.初始标记：STW，伴随young GC发生，标记从GC Root开始可达的对象

        1.单线程、全局扫描堆然后标记清除代价昂贵。

            2.2 增加并发周期的线程数量；

            2.3 让并发周期尽早开始，这个是通过设置堆使用占比来调整的（默认45%老年代占据整个堆的大小比例）

            2.4 在混合垃圾回收周期中回收更多的老年代区块；

优点：

    1.解决了CMS中内存碎片的问题。

    2.能够在指定时间片内优先回收价值最大的Region。

5.哪些对象会进入到老年代？

1.大对象。对Serial和parNew收集器而言，参数-XX:PretenureSizeThreshold代表对象大小如果大于它的值，则直接分配到老年代。

2.年龄达到阈值（默认15岁）。Minor GC 进入到Survivor，此时年龄为1，继续存活年龄将不断增加。

3.动态年龄判定。Surviovr区中年龄相等的对象占了整个Survivor大小的一半以上，则年龄大于或等于该年龄的对象将进入老年代。

4.空间分配担保。如果晋升到老年代的对象的平均大小 小于 老年代的剩余空间则发生担保，Eden内的所有存活对象进入到老年代，否则担保失败，发生Full GC.(比Minor GC 慢10倍)

public class Observable {
    private boolean changed = false;
    private Vector<Observer> obs;

    public Observable() {
        obs = new Vector<>();
    }

//保证线程安全
public synchronized void addObserver(Observer o) {
    if (o == null)
        throw new NullPointerException();
    if (!obs.contains(o)) {
        obs.addElement(o);
    }
}

public synchronized void deleteObserver(Observer o) {
    obs.removeElement(o);
}

public void notifyObservers() {
    notifyObservers(null);
}

public void notifyObservers(Object arg) {

    //一个临时的数组，用于并发访问被观察者时，留住观察者列表的当前状态
    Object[] arrLocal;//线程私有

    //注意这个同步块，它表示在获取观察者列表时，该对象是被锁定的
    //也就是说，在我获取到观察者列表之前，不允许其他线程改变观察者列表
    synchronized (this) {

        if (!changed)
            return;
        arrLocal = obs.toArray();
        //将改变标识重新置回未改变
        clearChanged();
    }

    //注意这个for循环没有在同步块，此时已经释放了被观察者的锁，其他线程可以改变观察者列表
    //但是这并不影响我们当前进行的操作，因为我们已经将观察者列表复制到临时数组
    //在通知时我们只通知数组中的观察者，当前删除和添加观察者，都不会影响我们通知的对象
    for (int i = arrLocal.length-1; i>=0; i--)
        ((Observer)arrLocal[i]).update(this, arg);
}

CopyOnWrite容器

发表于 2018-10-29 | 分类于 面试总结 ， java集合 | 阅读次数:

含义：

即写时复制的容器，往一个容器添加元素的时候，不直接往当前容器添加，而是先将当前容器进行Copy，复制出一个新的容器，
然后再往新的容器里添加元素，添加完元素之后，再将原容器的引用指向新的容器。

COW与ReentrantReadWriteLock的区别与联系？

相比于ReentrantReadWriteLock的特性：写锁被写线程获取后，读写线程都会被阻塞。而COW读线程不会被阻塞，性能又提升了。

不同点：

读写锁：读线程任何时候都是获取到最新的数据，满足数据实时性。原因在于为了实现数据实时性，
在写锁被获取后，读线程会等待或者当读锁被获取后，写线程会等待，从而解决“脏读”等问题。

COW: 牺牲数据实时性满足数据的最终一致性。即读线程对数据的更新是延时感知的，因此读线程不会存在等待的情况。

相同点：

1. 两者都是通过读写分离的思想实现。

2. 读线程间是互不阻塞的。

CopyOnWriteArrayList源码分析

写操作：需要加锁。

//CopyOnWriteArrayList.java
public boolean add(E e) {
    final ReentrantLock lock = this.lock;
    lock.lock();//保证多线程环境中只出现一个集合副本
    try {
        Object[] elements = getArray();
        int len = elements.length;

        //产生副本数组
        Object[] newElements = Arrays.copyOf(elements, len + 1);
        newElements[len] = e;
        setArray(newElements);//原引用指向新的容器
        return true;
    } finally {
        lock.unlock();
    }
}

//注意：private transient volatile Object[] array
//volatile保证可见性，在进行写时看到的已经是最新的数据的，但是只读的话就有可能是未更新的数据。
final void setArray(Object[] a) {
    array = a;
}

读操作：不需要加锁，存在并发读，读到的可能是旧数据。

public E get(int index) {
    return get(getArray(), index);
}

适用场景：

  • CopyOnWriteArrayList适合使用在读操作远远大于写操作的场景里，比如缓存。

缺点

1.内存占用问题。

2.数据一致性问题。
