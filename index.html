LRU的设计

发表于 2019-08-05 | 分类于 面试总结 ， 操作系统 | 阅读次数:

基于双向链表 + HashMap实现的LRU？

链表节点：

public class LruNode<K,V> {
    K key;
    V value;
    LruNode<K,V> before;
    LruNode<K,V> after;

    public LruNode(K key, V value, LruNode<K, V> before, LruNode<K, V> after) {
        this.key = key;
        this.value = value;
        this.before = before;
        this.after = after;
    }
}

LRU的put和get实现：

public class MyLRU<K,V> {
private HashMap<K,LruNode> map = new HashMap<>();
private LruNode head;
private LruNode tail;
private int capacity;

public MyLRU(int capacity) {
    this.capacity = capacity;
}

public void put(K key, V value){
    LruNode node = map.get(key);
    if(node == null){
        //链表不存在key节点,插入尾部
        insertNewNode(key,value);

        //实现缓存功能
        implementCache();
    }else{
        //已存在 因为访问了又将它放到链表尾部
        afterAccessNode(node);
    }
}

public V get(K key){
    LruNode node = map.get(key);
    if(node != null){
        afterAccessNode(node);
        return (V) node.value;
    }
    return null;
}

public void afterAccessNode(LruNode node) {
    LruNode last = tail;
    if(node != last){
        LruNode p,before,after;
        p = node;
        before = p.before;
        after = p.after;
        p.after = null;
        if(before == null){
            head = after;
        }else {
            before.after = after;
        }

        if(after != null){
            after.before = before;
        }else {
            last = before;
        }

        if(last != null){
            last.after = node;
            node.before = last;
        }else {
            head = node;
        }
        tail = p;
    }
}

public void implementCache() {
    if(map.size() > capacity) {
        //删除头节点
        if (head != null) {
            map.remove(head.value);//删除
            LruNode node = head.after;
            head.before = head.after = null;//help GC
            head = node;
            if (node != null) {
                //别忘了!!!
                node.before = null;
            }else {
                tail = null;
            }
        }
    }
}

public void insertNewNode(K key, V value) {
    LruNode newNode = new LruNode(key,value,null,null);
    LruNode last = tail;
    if(last != null){
        last.after = newNode;
        newNode.before = last;
    }else {
        head = newNode;
    }
    tail = newNode;
    map.put(key,newNode);
}

public void print(){
    LruNode cur = head;
    while (cur != null){
        System.out.print(cur.value + "->");
        cur = cur.after;
    }
}
}

redis底层数据结构

发表于 2019-07-26 | 分类于 Redis | 阅读次数:
  • SDS(Simple Dynamic string)（String类型）

free：buf中剩余可用空间的长度；
len：buf中已占用空间的长度；
buf：数据空间

优点：

1.O(1)时间获取字符串长度；

2.通过验证free的长度，解决了缓冲区溢出的问题；

3.减少了修改字符串长度造成的内存溢出；

    3.1 预分配

        如果对SDS修改后，len < 1M，那么len=2*len+1，1字节用来存'\0'；

        如果对SDS修改后，len > 1M，那么len=1M+len+1；

    3.2 惰性释放

        缩短字符串长度，redis不马上减少SDS所占内存，只是增加free的长度；

4.二进制安全。不依赖'\0'作为结束标志，而用len的长度标记字符串的长度；

5.兼容C语言。可用C的字符串API；
  • 链表

特性：

1.双向链表实现，获取链表长度的时间复杂度O(1)，可以自己实现dup、free
  和match函数；

2.有专门的迭代器实现迭代操作；
  • 字典(Hash类型)

字典节点：

typedef struct dictEntry {
    void *key;  //键

        void *val;
        uint64_t u64;
        int64_t s64;

Hash表存放字典节点：

typedef struct dictht { // hash表
    dictEntry **table;  // 节点数组
    unsigned long size; // hash表大小
    unsigned long sizemask; // hash表掩码，等于size-1，用于计算hash值
    unsigned long used; // 已有节点数量

字典：

typedef struct dict {  //字典
    dictType *type; //各种字典操作方法
    void *privdata; //私有数据，用于传给操作函数
    dictht ht[2]; //两个hash表，一个用来存储当前使用的，一个用来rehash
    long rehashidx; //rehash标志位，0表示rehash
    int iterators;  //迭代器的运行数量
} dict;

总结：

1.hit[0]存放数据，hit[1]在rehash时使用；

2.扩容和收缩。

  2.1 扩容条件：

         1.服务器没有执行bgsave或bgrewriteaof命令，loader >= 1;
        loader=ht[0].used/ht[0].size

      2.服务器目前正在执行bgsave或bgrewriteaof命令，
        并且哈希表的负载因子>=5

  2.2 收缩条件

      服务器没有执行bgsave或bgrewriteaof命令，
      并且负载因子<=0.1

  2.3 扩容大小

      大于等于 ht[0].used*2 的最小2的n次方

  2.4 收缩大小

      大小等于 ht[0].used 的最小2的n次方

3.渐进式hash

  分多次、渐进式地将ht[0]的键值对慢慢rehash到ht[1]。

  3.1 Dict同时持有ht[0]和ht[1]，rehashidx=0代表rehash开始；

  3.2 rehash时，每次对字典进行添加、删除、查找或更改，除了执行指定
      操作还会将 ht[0] 哈希表中在 rehashidx 索引上的所有键值对，
      rehash 到ht[1]，完成后rehashidx加1；

  3.3 delete、find、update在ht[0]和ht[1]上进行，add只在ht[1]上
      进行，保证ht[1]只增不减；

  3.4 ht[0]的所有键值对rehash到ht[1]后，free ht[0]，ht[0] 指向
      ht[1]，新建空白ht[1]
  • 整数集合(intset)

特性：

1.内存连续、数值存储有序、无重复；

2.三种编码方式，encoding表示一个元素所需的内存大小，有2字节、
  4字节、8字节的取值

NIO详解

发表于 2019-07-18 | 分类于 面试总结 ， 操作系统 | 阅读次数:

阻塞/非阻塞 和 同步/异步？

1.同步/异步：同步是会阻塞请求进程的，异步不会阻塞请求进程的，所以BIO、NIO
            和IO多路复用都是同步的；


2.阻塞/非阻塞：在等待一件事情的处理结果时，如果你还去干点其他的事，则为
  非阻塞，否则为阻塞；

阻塞IO和非阻塞IO？

1.阻塞IO

分析：

1.两个阶段都是阻塞的。第一个阶段指等待网络数据到达内核空间缓存区域，第二个阶段
  是将内核空间缓冲区的数据复制到用户进程缓冲区；

2.调用recvfrom时就开始阻塞；

2.非阻塞IO

分析：

1.第一个阶段是非阻塞的，第二个阶段是阻塞的；

2.调用recvfrom时，如果数据没有准备好，是不会阻塞进程的，应用进程会轮询内核，检查数据是否准备好，
准备好以后，数据复制到应用进程缓冲区，这段时间进程是被阻塞的；

3.IO多路复用

含义：

    指内核一旦发现进程指定的一个或多个IO条件准备就绪时，就通知该进程；
    复用同一个线程，来并发交替的访问多个描述符的IO操作；

好处：

    单个进程可以同时处理多个网络连接的IO；

    单线程占用资源少、不消耗太多CPU；

分析：

1.进程阻塞于select，内核监视所有select负责的套接字，任何一个套接字数据准备好
  以后，select方法返回；

2.一个进程能同时等待多个文件描述符，它们中的任意一个进入读就绪状态时，select()函数就可以返回；

3.第一个阶段和第二个阶段都阻塞；

4.在多路复用模型中，对每一个socket都设置成non-blocking，第一个阶段是被select
  阻塞的，而不是被socket IO阻塞的；

4.异步IO

1.aio_read后立马返回，不会阻塞用户进程，kernel等待数据准备完成，然后将数据
拷贝到用户内存，完成以后给用户进程发一个signal，告诉它read操作完成；

BIO、NIO、AIO的区别？

1.BIO：进程从调用recvfrom开始就阻塞；

2.NIO：进程大部分时间都不会block，但仍然要求进程去主动check，当数据准备完成
以后，也需要进程主动去调用recvfrom，将数据拷贝到用户空间；

3.AIO：用户进程完全不会阻塞，将整个IO操作交给kernel完成，它完成以后发信号
       通知，在此期间用户进程不需要检查IO操作的状态，也不需要主动拷贝数据；

同步IO、异步IO的区别？

1.同步IO：导致请求进程阻塞，直到I/O完成，其中阻塞IO、非阻塞IO、IO多路复用都是
  同步IO；

2.异步IO：不导致请求进程阻塞；

IO和NIO的区别？

1.前者面向流的，后者面向缓冲区的；

2.前者是阻塞的，后者非阻塞的；

    从硬盘读数据，程序一直等着，数据读完以后，继续操作，这叫阻塞IO；

    从硬盘读数据，然后程序继续向下执行，等数据读完以后，通知当前程序，
    然后此程序立即处理数据，这叫非阻塞IO；

3.后者selector单线程处理多个通道；

Selector的Reactor模型？

1.Dispatcher：

1.1 持有Synchronous Event Demultiplexer对象(理解为Selector对象)和
 CHM Handler和ConcreteHandler的映射集合，Handler可以包装成事件，所以CHM
也可以表示EventType和EventHandler的映射集合；

1.2 内部dispatch方法调用Selector的select方法，若无事件则阻塞，否则获取
    事件列表，轮询获取ConcreteHandler来处理每个Handler；


2.Selector：内部维护一个阻塞队列，队列为空时，调用select()方法阻塞，不为空
返回事件列表；

3.Acceptor：内部持有Selector对象，用来接收客户端的连接事件，并添加到selector对象中，
之后再通过Dispatcher中具体的ACCEPT EventHandler将事件的状态置为READ；

单线程的Reactor模式

注意：

1.单线程是针对所有的I/O操作，例如accept()、read()、write()以及connect（）等
  都是在一个线程上完成的；

2.可以将非IO操作的业务逻辑从Reactor单线程中卸载，来加速Reactor线程对IO请求的响应；

3.不适合高并发、高负载的应用场景；

多Reactor线程模式

注意：

1.mainReactor：注册一个Acceptor事件处理器到mainReactor中，监听客户端的ACCEPT事件，
并将事件派发给acceptor；

2.acceptor：处理器通过调用accept()方法获取客户端的SocketChannel，并将它
  传递给subReactor线程池；

3.subReactor：负责和客户端通信，可以开多个subReactor线程来处理海量请求；

4.所有的I/O操作还是在Reactor线程中完成的；

select、poll、epoll的区别？

重要区别：

1.select和poll动作基本一致，只是poll的fd用链表存储，select的fd用fd标注位存放；

2.select和poll不会明确指出哪些文件fd就绪，都需要通过遍历整个文件fd才能找到，
  epoll直接处理就行了；

3.select和poll需要将文件fd的数据结构拷贝进内核，最后在拷贝到用户空间。
  epoll中是将epoll_wait中返回的数据拷贝到用户空间；

4.select和poll采用轮询的方式来检查文件fd是否处于就绪状态，epoll采用回调机制，
  回调机制：
     执行epoll_ctl的add操作时，会注册回调函数，内核在检测到某文件描述符可读/
  可写时会调用回调函数，该回调函数将文件fd放在就绪链表中；

  当fd数量剧增后，
  select和poll的效率会线性下降，epoll不会受太大影响，除非活跃的socket很多；

epoll高效的原因：

    1.减少了用户态和内核态之间文件fd的拷贝；
      (epoll_ctl中会将epoll_event结构体从用户空间拷贝到内存空间)

    2.减少了对就绪文件fd的拷贝；

总结：

1.select

缺点：

    1.1 单个进程所打开的fd是有一定数量限制的，32位默认1024，64位默认
        2048；

    1.2 对socket进行扫描是线性扫描，效率低；

    1.3 用户空间和内核空间传递fd数据结构开销大；

2.poll

优点：
    2.1 基于链表存储，没有fd数量限制；

缺点：
    2.1 用户空间和内核空间传递fd数据结构开销大；

    2.2 水平触发，报告了的fd如果未处理，下次poll时会再次报告该poll；

3.epoll

特点：

    3.1 支持水平触发和边缘触发，最大特点在于边缘触发，它只告诉你哪些刚
        刚变为就绪，而且只会通知一次。

    3.2 使用事件的就绪通知方式，通过epoll_ctl注册fd，一旦fd就绪，内核
        会采用类似callback的回调机制来激活该fd，epoll_wait就可以收到
        通知；
        去掉了遍历文件描述符，而是通过监听回调的机制；
优点：

    3.1 没有最大并发连接数的限制；

    3.2 不是轮询的方式扫描socket，只有活跃的fd才会调用callback函数，
        epoll只管活跃的连接，跟连接总数无关；

    3.3 使用mmap减少复制开销；

区别：

1.支持一个进程所能打开的最大连接数。

    select有限，poll和epoll没有限制。

2.fd剧增后带来的IO效率问题。

    select和poll的"线性下降性能问题"，epoll是根据每个fd上callback函数
    来实现的，只有活跃的socket才会主动调用callback。

3.消息传递方式。

    select和poll中内核需要将消息传递到用户空间，都需要内核拷贝；

    epoll通过内核和用户空间共享一块内存实现的，返回的文件描述符是通过
    mmap(内核和用户空间的共享内存)实现传递的，较少了不必要的拷贝；

3.epoll_wait方法

  3.1 将readylist中的epitem出列，触发的事件从内核空间拷贝到用户空间，之后
      判断epitem是否需要重新添加到readylist中去；

  3.2 epitem重新添加到readylist中的情况

     3.2.1 epitem上有用户关注的事件触发；

     3.2.2 epitem被设置成水平触发LT；

  3.3 epitem被添加到readylist中的情况

     3.3.1 对fd调用EPOLL_ADD，这个fd上有用户关注的激活事件，则这个fd会
           被添加到readylist中；

     3.3.2 对fd调用EPOLL_MOD，新增了fd的关注事件且对应fd有相应的事件激活
           fd会被添加到readylist中；

     3.3.3 fd上有事件触发会调用ep_poll_callback回调函数，如果触发的事件
           是用户关注的事件，fd会被添加到readylist中；

4.LT和ET模式；


4.1 LT和ET工作机制

    如果是ET，epitem是不会再进入到ready_list中，除非fd再次发生了状态改变
ep_poll_callback被调用，
    如果是非ET，不管你还有没有有效的事件或者数据，都会被重新插入到ready_list
下次再调用epoll_wait时会立即返回，并通知给用户空间，如果这个被监听的fd
确定没事件也没数据了，epoll_wait会返回一个0；

有序性

JMM制定的volatile重排序规则

通过在volatile变量的操作前后插入内存屏障的方式，来禁止指令重排，进而保证多线程
间共享变量的有序性；

具体屏障：

    1.1 每个volatile写操作前插入StoreStore屏障，写操作之后插入StoreLoad屏障；

    1.2 每个volatile读操作前插入LoadLoad屏障，读操作之后插入LoadStore屏障；

可见性

1.写入一个volatile变量时，JMM会将线程本地内存的共享变量刷新到主内存；

2.读一个volatile变量时，JMM会把线程本地内存中的共享变量置为无效，直接从主存内读取共享变量；

补充：

volatile 修饰long和double变量，可以解决64位数据读写操作划分为两次32位的操作
产生的非原子性协定问题；

可以使long和double的赋值是原子的；

3.happens-before原则（8个）：

    3.1 程序次序规则：单线程内顺序发生；

    3.2 锁定规则：unlock操作先于对同一个锁的lock操作；

    3.3 volatile规则：对volatile变量的写操作先于读操作；

    3.4 传递规则:A先于B，B先于C，则A先于C；

    3.5 线程启动规则：Thread对象的start()先于该线程的每一个动作；

    3.6 线程中断规则：对线程interrupt()方法的调用先行发生于被中断线程的代码检测到中断事件的发生；

    3.7 线程终结规则：线程中所有的操作都先行发生于线程的终止检测，我们可以通过Thread.join()方法结束、Thread.isAlive()的返回值手段检测到线程已经终止执行；

    3.8 对象终结规则：一个对象的初始化完成先行发生于他的finalize()方法的开始；

2.缓存雪崩

2.1 什么是缓存雪崩？

    指所有key的缓存时间相同，在同一时间失效，或者缓存服务器宕机，导致所有请求同时访问数据库，从而使系统崩溃；

2.2 解决办法？

    1.交错失效时间。进行缓存失效时间设置时，在某个基础时间值上加上两秒内的随机数；

    2.增加互斥锁。集群环境下使用分布式锁(setnx + expire)，对于每个key，让一个线程构建缓存，其他线程等待，
    之后重新从缓存中获取数据；

    3.提高缓存的HA。

3.缓存击穿

3.1 什么是缓存击穿？

    缓存雪崩的一个特例，缓存雪崩是对于全部数据，而缓存击穿是针对热点数据。

3.2 解决办法？

    1.对于热点数据进行二级缓存，并对于不同级别的缓存设置不同的失效时间。

    2.本地缓存（集群不可用时） + 限流（限制请求量）

4.1 解决办法？

    1.设置不同失效时间；

中介者模式

 使用一个中介对象来封装一系列的对象交互，使得各对象间不需要显示地相互引用，
中介者模式的本质是：封装交互；

Colleague角色：抽象交互对象

//持有中介者的引用
public abstract class AbstractDatabase {
    protected AbstractMediator mediator;
    public static final String REDIS = "redis";
    public static final String MYSQL = "mysql";
    public static final String HDFS = "hdfs";

    public AbstractDatabase(AbstractMediator mediator) {
        this.mediator = mediator;
    }

    abstract void addData(String data);
}

ConcreteColleague角色:具体交互对象

public class RedisDatabase extends AbstractDatabase {
    private List<String> redisSet = new ArrayList<>();
    public RedisDatabase(AbstractMediator mediator) {
        super(mediator);
    }

    public void synInfo(String data){
        System.out.println("redis data is synchronizing......");
        //交由中介者实现数据同步
        mediator.sync(AbstractDatabase.REDIS,data);
        System.out.println("redis data finished synchronize");
    }

    @Override
    void addData(String data) {
        System.out.println("add data to Redis");
        redisSet.add(data);
        synInfo(data);
    }

    public void redisInfo(){
        System.out.println("stored REDIS data number: "+redisSet.size() + " Info : "+redisSet.toString());
    }
}

    }
