JVM调优

发表于 2019-08-23 | 分类于 面试总结 ， java虚拟机 | | 阅读次数:

1.虚拟机参数

-Xms：最小堆空间

-Xmx：最大堆空间

-Xmn：设置年轻代的大小

-Xss：每个线程占用的栈内存大小，整个虚拟机栈可分配的内存跟物理机的
      内存大小、Java堆内存、方法区内存大小相关


-XX：+PrintGCDetails：输出GC的详细日志

-XX：MetaspaceSize：元空间的大小

-XX：CMSInitiatingOccupancyFraction：老年代空间达到70%就开始执行
                            CMS，老年代有足够的空间接收年轻代对象

-XX：SurvivorRatio：Eden区与Survivor区的大小比值，例如：8代表
                    两个S区和E区的比值为：2:8即1:1:8

-XX:MaxTenuringThreshold：对象从年轻代晋升到老年代的年龄阈值，默认15；

a)参数基本策略

    活跃数据大小：程序稳定运行期间，长期存活在堆中的对象大小，即Full GC
                 后堆中老年代占用空间大小，通过查看Full GC日志获得。


b)优化

    1.扩容新生代容量。扩容以后虽然会增加Eden区的扫描时间，但是由于Minor 
      GC的间隔增加了，对象的生命周期短的就不用在进行复制了，显然对象的
      复制成本是要高于扫描Eden区的成本的；

    2.CMS中的remark阶段比较耗时，因为要扫描全堆(新生代对象引用指向老年代)
      所以可以在remark发生之前增加一次 Minor GC，减少要扫描的对象。

      eg:增加CMSScavengeBeforeRemark参数，remark阶段前强制进行一次
         Minor GC

    3.将 MaxPermSize和MinPermSize设置成一致、Xms =Xmx，可以减少内存
      自动扩容和收缩带来的性能损失。

2.命令行工具

a）jps：查看虚拟机进程状态

b) jstat：虚拟机统计信息监视工具

   参数：

     -class：监视类装载、卸载数量、总空间以及类装载消耗时间

     -gc：监视Java堆状况、包括eden、survivor等的容量、已用空间、
          GC时间合计等；

     -gcutil：与gc类似，输出已用空间占总空间的占比；

c) jinfo：查询Java虚拟机配置信息

d）jmap：生成堆转储快照，称为heapdump或dump文件

    -dump：生成Java转堆快照。

        eg：jmap -dump:format=b，file=filename 3500

    -heap：Linux/Solaris平台下有效，显示堆详细信息，如使用哪种回收器
           参数配置、分代状况等等；

e) jhat：分析jmap生成的dump文件，很少用；

f) jstack：用于生成虚拟机当前时刻的线程快照，线程快照：即每一条
           线程正在执行的方法堆栈的集合；

3.问题定位步骤

a) 定位进程号，jstat监视堆状况

    jstat -gc 11172 1000 //11172进程号，1000每隔1000ms打印一次日志

b) jmap把当前堆dump下来

    jmap -dump:live,format=b,file=dump.hprof 11172

    使用 VisualVM 打开dump.hprof 文件进行分析

4.内存泄露问题

a) 非静态内部类(匿名内部类)导致内存泄露

   非静态内部类会持有外部类的引用，导致GC可能回收不了这部分引用，导致
   OOM

       eg:

        public class NonStaticInnerDemo {
          //注意是static
          //生命周期比NonStaticInnerDemo长
          private static String TAG = "Outter";

          //private static Runnable runnable 未持有外部引用
          private Runnable runnable = new Runnable(){
            public void run(){
              System.out.println("inner run: " + TAG);
            }
          }
        }
    看一下编译后的代码：
        putfield  #1// Field this$0:LNonStaticInnerDemo;
     表示有一个对NonStaticInnerDemo的引用存在了，即持有了外部类的
    引用

    静态的内部类不会持有外部类的引用。
