java虚拟机垃圾收集器

发表于 2018-08-15 | 分类于 java虚拟机 | | 阅读次数:

垃圾回收前先标记,标记的方法？

1.引用计数

给对象添加一个引用计数器，每一个地方引用它时，计数器值就加1.

缺点：无法检测出循环引用。

2.可达性分析

不在”GC Roots”引用链上的对象会被标记为将要收集的对象。

可作为GC Roots的对象：

  • 虚拟机栈（栈中的本地变量表）中引用的对象

  • 方法区中类静态属性引用的对象

  • 方法区中常量引用的对象

  • 本地方法栈（native方法）引用的对象、

垃圾收集算法

  • 标记-清除算法

    1.清除不在从GC ROOTs根路径扫描中的对象。

    2.缺点是会产生内存碎片。

  • 标记-整理算法

    1.清除完上述算法中未被标记的对象后，让所有存活的对象向一端移动，然后直接清理掉端边界以外的内存，解决了内存碎片问题，当然开销也变大了。

    2.适用于老年代垃圾的回收。

  • 复制算法

    1.将内存区域分为对象区域和空闲区域，将所有存活的对象都复制到空闲区域。

    2.需要额外的内存空间进行分配担保。不适用于对象存活率较高的老年代。

    3.额外空间可以向老年代借。老年代空间都还不够用的话，”Stop the world”发生Full GC。Full GC的作用是为了让老年代腾出空间来。

  • 分代回收算法

    1.将内存划分为新生代和老年代，新生代由于“朝生夕死”的特点，即对象的存活率低，可采用复制算法；老年代对象存活率高，没有额外内存空间担保，所以采用标记-清除算法和标记-整理算法较好。

垃圾收集器

HotSpot虚拟机的垃圾收集器：

默认的垃圾收集器：Serial收集器 / Serial Old 收集器

  • Serial收集器（复制算法）

    1.在收集垃圾时，必须暂停其它所有工作的线程，直到它收集结束。

    2.没有线程交互的开销，效率最高的单线程垃圾收集器。

    3.适用于运行在Client端的虚拟机

  • ParNew收集器（停止-复制算法）

    1.多线程。

    2.适用于运行在Server端的虚拟机。

    3.唯一一个能与CMS收集器（老年代的收集器）配合使用。

  • Parallel Scavenge收集器（停止-复制算法）

    1.吞吐量优先的收集器。

  • Serial Old 收集器（标记-整理算法）

  • Parallel Old 收集器（停止-复制算法）

    1.在吞吐量和CPU资源敏感的环境下，与Parallel Scavenge配合使用。

  • CMS收集器（标记-清理算法）

    1.过程：

    a.初始标记

b.并发标记

c.重新标记

d.并发清除

    2.初始标记和初始标记依然需要停顿。

    3.缺点：

    a.应用程序变慢,吞吐量变低

b.无法处理浮点动垃圾

c.会产生太多空间碎片

  • G1 收集器(得完整看看)

    1.使用Region划分内存空间以及有优先级的区域回收。

    2.G1在扫描了region以后，对其中的活跃对象(GC Roots上的对象)的大小进行排序，首先会收集那些活跃对象小的region，以便快速回收空间（要复制的活跃对象少了）

    特点：

    a.并行和并发。

b.分代收集。

c.空间整合。

d.可预测的停顿。
